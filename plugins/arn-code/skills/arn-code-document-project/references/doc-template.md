# Documentation Template

This template defines the structure for project documentation generated by `arn-code-document-project`.

## Principles

1. **Truth over plan** — Document what was built, not what was planned. If implementation diverged from the plan, document reality.
2. **Concise** — Target 150-400 lines. Every sentence must add value.
3. **Reference-heavy** — Point to files with exact paths. Don't duplicate code in the docs.
4. **LLM-friendly** — Use tables, bullet lists, and consistent headings. Include exact class/function names.
5. **Human-friendly** — Explain the "why" behind key decisions. Include usage examples for complex features.
6. **Accurate** — Cross-reference the code. Don't document features that don't exist.
7. **Language-agnostic** — The template works for any programming language or framework.

## Template

Every section is required unless explicitly marked optional.

### Header

```markdown
# [Feature/Fix Name]

**Status**: Production Ready | Partial | In Progress
**Date**: [Date from reports or current date]
**Spec**: [spec file path, if available]
```

### Overview

2-3 sentences: what this feature/fix does, why it exists, what problem it solves. Pull from the spec's problem statement if available.

### Architecture

Concise description of how components fit together. Include an ASCII diagram if the feature has 3+ interacting components.

#### Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| ... | ... | ... |

Pull from spec decisions log and INTRODUCTION.md. Note any decisions that changed during implementation.

### Key Components

| Component | Path | Purpose |
|-----------|------|---------|
| ... | `path/to/file` | ... |

List all files created or significantly modified. Use exact paths from reports and git diff.

### Usage

Include relevant subsections based on what was built:
- **API Endpoints** — method, endpoint, auth, description (table format)
- **Service Methods** — key public methods with signatures
- **CLI Commands** — if applicable
- **Configuration** — settings, env vars, defaults (table format)

Only include subsections that apply.

### Integration Points

How this feature connects with other parts of the system. Be specific — name classes, methods, and modules.

### Testing

- **Test files**: list with paths
- **Total tests**: N (N passing, N failing, N skipped)
- **Run command**: exact command to run the tests

Pull from testing reports.

### Implementation Notes (optional)

Only include if there are noteworthy items:

#### Plan Divergences
What changed from the original plan and why. Based on divergence analysis from Step 5.

#### Known Limitations
Only document actual limitations, not hypothetical ones.

#### Bugs Fixed During Implementation
Notable bugs found and fixed during testing. Pull from `bugsFixed` in testing reports.
